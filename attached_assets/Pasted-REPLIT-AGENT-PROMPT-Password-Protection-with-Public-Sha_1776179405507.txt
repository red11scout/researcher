REPLIT AGENT PROMPT — Password Protection with Public Shareable Reports
WHAT THIS APP NEEDS
Add password protection to this entire application while keeping shared report links publicly accessible. People who receive shared report links must NOT be able to access any other part of the app.
STEP 1: ADD ENVIRONMENT VARIABLES
Add these Replit Secrets (Environment Variables):

APP_PASSWORD — set to a strong password (I will set this manually after you create the variable)
SESSION_SECRET — generate a random 64-character hex string and set it automatically

STEP 2: INSTALL DEPENDENCIES
Install these packages:

cookie (for parsing cookies)
jsonwebtoken (for signing session tokens)
bcryptjs (for hashing the password comparison — optional but recommended)
express-rate-limit or equivalent if using Express; if using Next.js, use rate-limiter-flexible

STEP 3: CREATE THE LOGIN PAGE
Create a login page at route /login. This page must:

Show a centered card on a clean white/dark background
Display the app title at the top: "BlueAlly AI Assessment"
Have a single password input field (type="password") with placeholder "Enter password"
Have a "Sign In" button below the input
On form submit, POST to /api/auth/login with { password: "<input value>" }
If the API returns success, redirect the browser to / (or to the URL in the returnTo query parameter if present, e.g., /login?returnTo=/report?company=Honeywell)
If the API returns an error, show "Invalid password" in red text below the input
After 5 failed attempts within 60 seconds, disable the button and show "Too many attempts. Please wait 60 seconds."
This page must be visually clean and professional. Use the app's existing styling/theme.
This page must be accessible WITHOUT authentication (it IS the auth entry point)

STEP 4: CREATE THE AUTH API ENDPOINTS
POST /api/auth/login
Request body: { "password": "string" }

Logic:
1. Compare request body password to process.env.APP_PASSWORD
2. If match:
   - Create a JWT token signed with process.env.SESSION_SECRET
   - Token payload: { authenticated: true, iat: <now>, exp: <now + 24 hours> }
   - Set an HTTP cookie named "session" with the JWT value
   - Cookie options: httpOnly=true, secure=true, sameSite="lax", path="/", maxAge=86400
   - Return { success: true }
3. If no match:
   - Return 401 { success: false, error: "Invalid password" }
POST /api/auth/logout
Logic:
1. Clear the "session" cookie by setting it with maxAge=0
2. Return { success: true }
3. Redirect to /login
STEP 5: CREATE AUTHENTICATION MIDDLEWARE
This is the most critical step. Create middleware that runs on EVERY request BEFORE any route handler.
Routes that MUST BE PUBLIC (no auth check, pass through immediately):
/login
/shared
/shared/*
/api/shared/*
/api/auth/*
/_next/*
/static/*
/assets/*
/public/*
/favicon.ico
/robots.txt
Any file with extensions: .js, .css, .png, .jpg, .svg, .ico, .woff, .woff2, .ttf, .map
Routes that MUST BE PROTECTED (require valid session cookie):
Everything else. This includes:
/
/report
/report?company=*
/dashboard
/api/* (except /api/shared/* and /api/auth/*)
All other pages and API routes
Middleware logic:
For every incoming request:
1. Check if the request path matches any PUBLIC route pattern → if yes, pass through immediately
2. If not a public route, read the "session" cookie from the request
3. If no cookie exists → redirect to /login?returnTo=<current path> (for page requests) or return 401 JSON (for /api/* requests)
4. If cookie exists, verify the JWT using process.env.SESSION_SECRET
5. If JWT is invalid or expired → clear the cookie, redirect to /login (for pages) or return 401 (for APIs)
6. If JWT is valid → pass through to the route handler
How to detect if a request is an API call vs a page navigation:
- If the path starts with /api/ → it's an API call → return JSON { error: "Unauthorized" } with status 401
- If the request Accept header contains "application/json" → it's an API call → return 401 JSON
- Otherwise → it's a page request → redirect to /login?returnTo=<encoded current URL>
STEP 6: SECURE THE SHARED REPORT PAGES
The existing /shared/[token] pages need to be hardened so they cannot be used to access the main app:
6A: Shared report page layout isolation
The shared report page (/shared/*) must:

Use a DIFFERENT layout/wrapper than the main app pages
NOT render any navigation bar, sidebar, menu, or header that links to app routes like /, /report, /dashboard
NOT include any <a href> or <Link> tags pointing to protected routes
NOT import or use any authentication context, user state, or app-level state management
Show ONLY: the report content itself, plus a minimal footer with "Powered by BlueAlly"
Optionally show a CTA button: "Get Your AI Assessment" linking to https://gofasterwithai.com (external, NOT the app)

6B: Shared report data fetching isolation
The shared report page must fetch its data ONLY from a dedicated endpoint:

The shared report page should call ONLY /api/shared/[token] to get its data
The /api/shared/[token] endpoint must read from the database and return ONLY the pre-saved snapshot data
This endpoint must NOT call any AI APIs, external data sources, or perform any live computation
This endpoint must accept ONLY GET requests
For invalid, expired, or revoked tokens: return 404 { error: "Not found" } — do NOT return 403 (that reveals the token format is valid)

6C: Shared report token security
Ensure the share tokens:

Are generated using crypto.randomBytes(18).toString('base64url') or equivalent (minimum 24 characters, 144 bits of entropy)
Do NOT contain company names, sequential IDs, user IDs, timestamps, or any decodable information
Are stored in the database alongside the full report snapshot data
Cannot be enumerated (rate limit the /api/shared/* endpoint to 30 requests/minute/IP, and return 404 for invalid tokens with rate limit of 10 failed lookups/minute/IP)
