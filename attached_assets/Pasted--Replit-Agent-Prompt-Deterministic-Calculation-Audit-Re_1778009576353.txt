Deterministic Calculation Rules

1. No LLM Math

LLMs may write narrative explanations only.

LLMs must never:

* Calculate ROI
* Calculate totals
* Calculate percentages
* Calculate scores
* Calculate rankings
* Calculate averages
* Calculate weighted scores
* Calculate financial values
* Estimate values
* Round values
* Generate chart data
* Generate JSON numeric fields
* Generate markdown numeric fields
* Generate report numbers

If an LLM is used, it must receive already-verified numbers as immutable context.

The prompt must explicitly say:

You are not allowed to calculate, infer, estimate, round, modify, or create numeric values. 
Use only the verified numbers provided in the structured calculation payload.
If a number is missing, say the number is unavailable.

2. No Floating-Point Drift

Do not use raw JavaScript floating-point arithmetic for business-critical calculations.

Avoid this pattern:

const result = value * percentage / 100

Use a deterministic decimal library or deterministic calculation engine.

Preferred options:

* decimal.js
* big.js
* dinero.js for money
* hyperformula for spreadsheet-like calculations
* A pure TypeScript deterministic calculation module with strict decimal handling

All currency, percentages, weighted scores, ratios, and totals must use deterministic decimal handling.

3. One Formula Registry

Create a centralized formula registry.

Example:

export const FORMULAS = {
  revenueImpact: {
    id: "revenue_impact_v1",
    description: "Revenue impact equals uplift percent times baseline revenue times gross margin percent.",
    formula: "(upliftPct / 100) * baselineRevenue * (grossMarginPct / 100)",
    inputs: ["upliftPct", "baselineRevenue", "grossMarginPct"],
    output: "revenueImpact",
    precision: 2,
    rounding: "ROUND_HALF_UP",
  },
};

Every calculation must have:

* Formula ID
* Formula version
* Input fields
* Output field
* Units
* Precision rule
* Rounding rule
* Validation rules
* Test cases
* Independent verification method

4. No Duplicate Calculation Logic

Search the entire codebase for duplicated math.

Remove or refactor any calculation logic found in:

* React components
* API handlers
* Report templates
* Markdown exporters
* JSON serializers
* Chart builders
* Prompt builders
* Shareable link renderers

Replace all duplicate logic with calls to the canonical calculation service.

Required Validation Layer

Before any calculation runs, validate all inputs.

Use strict schemas with Zod or a comparable validation library.

Every numeric input must define:

* Required vs optional
* Type
* Allowed range
* Unit
* Precision
* Default behavior
* Null behavior
* Error behavior
* Whether zero is allowed
* Whether negative values are allowed

Example:

const RevenueImpactInputSchema = z.object({
  baselineRevenue: z.number().finite().nonnegative(),
  upliftPct: z.number().finite().min(0).max(100),
  grossMarginPct: z.number().finite().min(0).max(100),
});

Invalid inputs must block calculation.

Do not silently coerce bad values.

Do not convert empty strings to zero unless explicitly defined and justified.

Do not allow NaN, Infinity, undefined, or malformed numeric strings into calculation logic.

⸻

Required Independent Verification

Every calculation must be verified by a second independent method before publishing.

Use one of these approaches:

Preferred Approach

Primary calculation engine:

* Canonical deterministic TypeScript calculation service using decimal.js

Independent verification engine:

* Separate formula implementation using a different path, such as:
    * big.js
    * HyperFormula
    * A pure reference implementation
    * A test-only Python decimal verifier
    * A second TypeScript module with no shared formula code

The verification engine must not simply call the same function again.

That is not independent verification.

Verification Rules

For every calculated result:

* Calculate using primary engine
* Calculate using independent verifier
* Compare results using strict tolerance rules
* If values match, mark as verified
* If values do not match, block publishing
* Log the mismatch
* Surface a clear error to the user or admin

Example result object:

type VerifiedCalculationResult = {
  formulaId: string;
  formulaVersion: string;
  inputs: Record<string, unknown>;
  primaryResult: string;
  verificationResult: string;
  verified: boolean;
  tolerance: string;
  units: string;
  precision: number;
  roundingMode: string;
  calculatedAt: string;
  inputHash: string;
  resultHash: string;
};

Rounding and Formatting Rules

Define one global rounding and formatting policy.

Do not allow different parts of the app to round differently.

Create a utility like:

formatCurrency(value, { precision: 2 })
formatPercent(value, { precision: 1 })
formatScore(value, { precision: 1 })
formatHours(value, { precision: 1 })

Rules must be explicit:

* Currency: 2 decimals unless displaying executive summaries, where compact formatting is allowed only from verified rounded values
* Percentages: define whether 25 means 25% or 0.25
* Scores: define scale, usually 1–10 or 0–100
* Hours: standardize all time values to hours
* Large numbers: use deterministic compact display formatting
* Never format by ad hoc string manipulation inside components

Store raw verified values separately from display-formatted values.
